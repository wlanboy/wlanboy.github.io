1 Grundlagen – Serving & Eventing

Knative besteht aus zwei unabhängig installierbaren Projekten auf Kubernetes:
Serving (request-getriebene, automatisch skalierende HTTP/gRPC-Workloads inkl. Scale-to-Zero) und
Eventing (event-getriebene, lose gekoppelte Kommunikation über das CNCF-Standardformat CloudEvents).

Knative Serving

Deployt Container als Service, versioniert jede Änderung als unveränderliche Revision und skaliert Pods nach Bedarf – bis auf 0 Instanzen.

Knative Eventing

CloudEvents-Producer und -Consumer werden über Broker und Trigger lose gekoppelt – Producer kennen ihre Consumer nicht.

Gemeinsames Fundament

Beide Projekte laufen auf normalen Kubernetes-Objekten (Deployment, Pod, Service) und teilen sich die Ingress-Implementierung am Mesh-Rand.

2 Knative Service – Configuration & Revision

Wie hängen Service, Configuration und Revision zusammen?

Ein Service (ksvc) ist die einzige Ressource, die Entwickler typischerweise anlegen. Intern verwaltet er eine Configuration (gewünschter Zustand: Image, Env, Resources) und eine Route. Jede Änderung an der Configuration erzeugt automatisch eine neue, unveränderliche Revision – alte Revisions bleiben erhalten und können weiter Traffic empfangen (Rollback, Canary).

Vom Service zu mehreren Revisions

apiVersion: serving.knative.dev/v1
kind: Service
metadata:
  name: shop-frontend
spec:
  template:
    metadata:
      name: shop-frontend-v2  # erzeugt Revision "shop-frontend-v2"
    spec:
      containers:
        - image: registry.example.com/shop-frontend:v2
  traffic:
    - revisionName: shop-frontend-v1
      percent: 90
      tag: current
    - revisionName: shop-frontend-v2
      percent: 10
      tag: candidate

3 Ingress / Networking Layer – der Eintrittspunkt

Was macht die Networking Layer?

Analog zum Istio Ingress Gateway: ein eigenständiges Envoy-Deployment am Cluster-Rand (Standardimplementierung Kourier, austauschbar gegen Istio- oder Contour-Ingress). Es terminiert externen Traffic (TLS), liest den Host-Header (z.B. shop-frontend.default.example.com) und leitet an die passende Route weiter.

4 Route – Traffic Splitting zwischen Revisions

Wofür ist die Route zuständig?

Bildet einen stabilen öffentlichen Hostnamen auf eine oder mehrere Revisions ab – inklusive prozentualer Aufteilung. Funktional das Knative-Gegenstück zu VirtualService + DestinationRule bei Istio, nur bereits im Service-Manifest über das traffic-Feld konfiguriert. Jede Revision erhält zusätzlich eine eigene, feste URL über tag – ideal, um eine Candidate-Version gezielt anzutesten, bevor sie Prozent-Traffic bekommt.

5 Autoscaling & queue-proxy – Scale-to-Zero

queue-proxy – der Sidecar jeder Revision

Jeder Revision-Pod bekommt einen queue-proxy-Sidecar injiziert (Knatives Gegenstück zum Envoy-Sidecar bei Istio). Er misst die aktuelle Concurrency (parallele Requests) und meldet sie laufend an den Autoscaler.

Autoscaler (KPA) & Activator

Der Autoscaler skaliert Pods anhand der gemeldeten Concurrency – auch auf 0. Fällt die Pod-Zahl auf 0, übernimmt der Activator als Platzhalter: er nimmt Requests entgegen, puffert sie und stößt das Hochskalieren an.

Cold Start vs. Warmer Pfad

  1. Warmer Pfad (Pods laufen bereits): Ingress → Route → Revision-Service → queue-proxy → user-container. Der queue-proxy meldet Concurrency, blockiert aber nicht.
  2. Cold Path (0 Pods): Ingress → Route leitet an den Activator, da keine gesunden Endpoints existieren.
  3. Activator puffert den Request und signalisiert dem Autoscaler Bedarf > 0.
  4. Autoscaler skaliert das Deployment der Revision von 0 auf 1+ Pods hoch.
  5. Sobald ein Pod (queue-proxy + user-container) bereit ist, leitet der Activator den gepufferten Request dorthin weiter und tritt danach wieder aus dem Datenpfad zurück.

6 Eventing Grundlagen – Broker & Trigger

Wofür sind Broker und Trigger zuständig?

Der Broker ist der zentrale Event-Eingang: Producer senden CloudEvents per HTTP POST, ohne ihre Consumer zu kennen. Ein Trigger ist ein Abonnement auf einem Broker – er definiert einen Filter (CloudEvent-Attribute wie type oder source) und einen Sink als Ziel (z.B. ein Knative Service).

Für die interne Skalierung eines Brokers (z.B. Kafka-backed, Multi-Tenant-Architekturen) siehe die vertiefende Seite zur hierarchischen Broker-Architektur.

Event-Fluss mit Filterung

apiVersion: eventing.knative.dev/v1
kind: Broker
metadata:
  name: default
  namespace: shop
---
apiVersion: eventing.knative.dev/v1
kind: Trigger
metadata:
  name: order-created-trigger
  namespace: shop
spec:
  broker: default
  filter:
    attributes:
      type: com.example.order.created
  subscriber:
    ref:
      apiVersion: serving.knative.dev/v1
      kind: Service
      name: order-processor

7 Source & Sink – externe Systeme anbinden

Source – Events von außen rein

Eine Source (z.B. PingSource, ApiServerSource oder eine RabbitmqSource) holt Events von einem externen System ab und wandelt sie in CloudEvents um, bevor sie an einen Broker gesendet werden.

Sink – Events nach außen raus

Ein Sink ist frei wählbar: Knative Service, Channel, Broker – oder direkt eine externe URI (z.B. ein Partner-Webhook oder ein externes RabbitMQ). Ein eigenes Egress-Gateway-Konzept kennt Knative dabei nicht: Netzwerkzugriff nach außen wird von der Plattform geregelt (z.B. NetworkPolicy, oder falls Istio parallel installiert ist, dessen Egress Gateway).

Externe Quelle rein, externe Senke raus

# Externe Quelle: Events von einem RabbitMQ außerhalb des Clusters abholen
apiVersion: sources.knative.dev/v1alpha1
kind: RabbitmqSource
metadata:
  name: external-rabbitmq-source
spec:
  rabbitmqClusterQueueName: orders-queue
  rabbitmqClusterHostAndPort: rabbitmq.partner.example.com:5671
  sink:
    ref:
      apiVersion: eventing.knative.dev/v1
      kind: Broker
      name: default
---
# Externe Senke: gefilterte Events direkt an eine externe URI zustellen
apiVersion: eventing.knative.dev/v1
kind: Trigger
metadata:
  name: forward-to-partner
spec:
  broker: default
  filter:
    attributes:
      type: com.example.order.shipped
  subscriber:
    uri: https://rabbitmq.partner.example.com/publish

8 Gesamtüberblick – kompletter Traffic-Pfad

Alle Bausteine zusammen: ein externer User ruft über die Ingress-Layer einen Knative Service auf, der aus dem Stand hochskaliert (Scale-to-Zero) und dabei ein CloudEvent auslöst, das über Broker und Trigger gefiltert und an ein externes RabbitMQ zugestellt wird.
