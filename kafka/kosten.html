Knative Eventing Auswirkungen auf Kafka

Was kostet ein globaler Broker wirklich?

Wenn viele Microservices über einen gemeinsamen Knative-Broker laufen, wächst die Kafka-Last nicht mit dem Nutzen, sondern mit der Zahl aller Trigger.

  1. 1 Empfangen Der Java-Receiver nimmt das Event per HTTP an und schreibt es einmal ins Kafka-Topic.
  2. 2 Lesen Pro Trigger eine Consumer-Group. Jede liest jedes Event aus dem Topic — noch ungefiltert.
  3. 3 Filtern Erst hier, im Java-Dispatcher, wird der Trigger-Filter ausgewertet. Treffer → weiter, sonst verworfen.
  4. 4 Zustellen Passende Events gehen per HTTP an den Ziel-Service.

Die Last in einer Formel

Global
Reads/s ≈ Events/s × Triggergesamt

Ein einziges, geteiltes Topic. Jeder Trigger im Cluster liest jedes Event.

Dezentral
Reads/s ≈ ΣBroker ( Events/s × TriggerBroker )

Jede Domäne trägt nur ihre eigene Last — verteilt über viele kleine Topics.

Beispielrechnung

1.000.000 Events/s, 50 Trigger. Global: bis zu 50.000.000 Reads/s auf einem Topic.   Dezentral: ~20.000 Reads/s
