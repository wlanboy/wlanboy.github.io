Hierarchische KNative Broker

1 KNative Eventing – Grundlagen

Broker

Zentraler Event-Sammelpunkt. Produzenten senden CloudEvents per HTTP POST. Der Broker nimmt sie an und verteilt sie an registrierte Trigger.

Trigger

Abonnement auf einem Broker. Definiert einen Filter (CloudEvent-Attribute) und ein Sink (Ziel: Service, anderer Broker, Channel).

CloudEvent

Standardformat (CNCF). Enthält Pflichtattribute: id, source, type, specversion – plus beliebige Extensions.

Event-Fluss (Basis)

2 Das Problem – Direkte Broker-Kette

Die intuitive Lösung für "global → namespace" sieht so aus: Ein globaler KNative Broker empfängt alle Events. Pro Namespace gibt es einen eigenen KNative Broker mit mehreren Triggern. Trigger auf dem globalen Broker leiten gefilterte Events an den jeweiligen Namespace-Broker weiter.

Broker Kette

Was passiert intern bei jedem Event:
  1. Event kommt an → wird in global-events Topic geschrieben (Kafka Schreibvorgang #1)
  2. Global Trigger liest aus global-events (Kafka Lesevorgang)
  3. Trigger prüft Filter → Event passt zu Namespace "alpha"
  4. Trigger sendet Event per HTTP an den Namespace-Broker
  5. Namespace-Broker schreibt Event in seinen Topic (Kafka Schreibvorgang #2)
  6. Namespace-Trigger liest aus dessen Topic und liefert an Service

Ergebnis: Ein Event wird mindestens 2× in Kafka gespeichert. Bei 3 Namespaces und einem Event das in 2 davon passt: 3× gespeichert (1× global-events + 1× ns-alpha + 1× ns-beta).
Mit Replikationsfaktor 3 multipliziert sich das weiter: jeder Kafka-Schreibvorgang erzeugt 3 physische Kopien –
also bis zu 9 physische Kopien eines einzelnen Events im Cluster.

3 Warum das ein ernstes Problem ist

Speicher-Explosion

Kafka speichert Events für die konfigurierte Retention (z.B. 7 Tage). Bei 10 Namespaces und 1 GB/Tag Eventvolumen:
10 GB/Tag × 7 Tage = 70 GB extra nur für Kopien.

Latenz-Verdopplung

Jeder Kafka-Hop kostet Zeit: Schreiben + Lesen + Polling-Interval + HTTP-Weiterleitung. In dieser Architektur: mindestens 2 vollständige Kafka-Zyklen bevor ein Service das Event sieht.

Kafka Cluster Druck

Doppelter Durchsatz auf dem Kafka-Cluster: alle Events zweimal geschrieben, zweimal gelesen.
Mit Replikationsfaktor 3 bedeutet das 6 physische Schreibvorgänge pro Event.
Bei hohem Volumen kann dies zu Broker-Überlastung und Rebalancing-Problemen führen.

Fehlerszenarien

Wenn der globale Broker-Trigger einen Namespace-Broker nicht erreicht (HTTP-Fehler), muss er erneut versuchen – während das Event bereits im globalen Topic liegt und sich Consumer-Lag aufbaut.

4 Lösungen – Optimierungsansätze

Ansatz I – Consumer-seitige Filterung (aktueller KNative-Stand)

KNative filtert ausschließlich consumer-seitig im Trigger: Alle Events landen im Kafka-Topic des Namespace-Brokers; der Trigger liest sie und verwirft nicht passende Events lokal.

Die Idee, Filter upstream zur Quelle zu propagieren (KNative Issue #934), wurde vom KNative-Team explizit als „wont-fix" geschlossen und ist nicht implementiert. Das CeSQl-Filtersystem wurde jedoch bewusst kompatibel dazu entworfen – die Filterdialekte stehen bereits bereit. 

# Filter-Dialekte die KNative Trigger unterstützt:
# 1. Legacy Exact-Match (spec.filter.attributes)
spec:
  filter:
    attributes:
      type: com.example.order.created
      source: team-alpha

# 2. Modern CeSQl-Ausdruck (spec.filters)
spec:
  filters:
    - cesql: "type LIKE 'com.example.order%' AND team = 'alpha'"

Ansatz II – Kafka-natives Partitions-Routing

Der Producer setzt beim Schreiben einen Partition Key = Namespace/Team-Name. Kafka verteilt Events damit deterministisch auf feste Partitionen. Jede KafkaSource-Instanz pro Namespace abonniert nur die ihr zugewiesenen Partitionen über explizite Consumer-Group-Partition-Assignments.

Das Routing findet auf reiner Kafka-Infrastruktur-Ebene statt – vollständig außerhalb von KNative Eventing. Kein Filtering, kein Broker-Hop, keine CPU für CEL-Auswertung.

Wann sinnvoll:

  • Namespaces sind stabil (selten neue Teams)
  • Producer-Code ist kontrollierbar
  • Kafka-Expertise im Team vorhanden
  • Sehr hohe Durchsatzanforderungen

Einschränkungen:

  • Partition-Count limitiert Namespace-Anzahl
  • Re-Partitionierung erfordert Topic-Neuanlage
  • Kein Attribut-basiertes Routing möglich
  • Routing-Logik liegt beim Producer, nicht bei KNative

5 Zu testender Lösungsansatz

Kombination aus Ansatz I + II: Ein einziges zentrales Kafka-Topic für alle Events, pro Namespace eine eigene KafkaSource mit dedizierter Kafka Consumer Group.
Aus Ansatz I kommt das Trigger-basierte Feinrouting – Filterung findet consumer-seitig im Namespace-Broker statt, kein globaler KNative Broker als Zwischenschicht.
Aus Ansatz II kommt die Kafka-native Isolation: jede Consumer Group verwaltet ihren Offset vollständig unabhängig, ohne Partition-Neuzuweisung oder Producer-Änderungen.

Wie der Ansatz funktioniert – Schritt für Schritt

  1. Alle Produzenten schreiben Events in ein zentrales Kafka-Topic (global-events) – keine Änderung am Producer erforderlich.
  2. Pro Namespace ist eine KafkaSource konfiguriert, die dieses Topic mit einer eigenen Consumer Group (z.B. cg-alpha, cg-beta) liest.
  3. Jede KafkaSource liest alle Events aus dem zentralen Topic und liefert sie an den namespace-lokalen Broker. Der Trigger filtert consumer-seitig – nur passende Events werden an Services weitergeleitet, nicht passende werden verworfen.
  4. Gefilterte Events werden direkt an den namespace-lokalen Broker geliefert – ohne globalen KNative Broker als Zwischenschicht.
  5. Innerhalb des Namespace routen Trigger die Events fein-granular an die zuständigen Services. Konfiguration liegt vollständig beim Namespace-Team.

Vorteile dieses Ansatzes

  • Events werden genau 1× in Kafka gespeichert – kein Topic-Duplikat, keine physischen Kopien durch Broker-Ketten
  • Volle Consumer-Group-Isolation: Rückstand oder Fehler in einem Namespace beeinflussen andere Namespaces nicht
  • Neue Namespaces ohne zentralen Eingriff: eine neue KafkaSource + Consumer Group genügt – Producer und zentrales Topic bleiben unverändert
  • Fein-Routing im Namespace durch Trigger – Teams definieren Filterregeln und Ziele selbstständig, ohne Abhängigkeit von einer zentralen Konfiguration
  • Nur 1 Kafka-Hop vom globalen Topic bis zum Service – minimale Latenz, kein zusätzlicher HTTP-Weiterleitung-Zyklus
  • Kein globaler KNative Broker als Single Point of Failure im Event-Fluss zwischen Producer und Namespace

Voraussetzungen & Risiken

  • Jede KafkaSource liest alle Events aus dem zentralen Topic – bei n Namespaces und hohem Volumen erhöht sich der Kafka-Leseaufwand linear mit der Namespace-Anzahl
  • Events ohne passenden Trigger-Filter bleiben unverarbeitet – eine Dead-Letter-Strategie für nicht konsumierte Events ist empfohlen
  • Anzahl aktiver KafkaSource-Instanzen skaliert linear mit Namespace-Anzahl – bei sehr vielen Namespaces den Connection-Overhead zum Kafka-Cluster im Blick behalten

6 Filter-System im Detail

KNative Trigger unterstützen mehrere Filter-Dialekte: von einfachem Exact-Match über Prefix/Suffix bis zum mächtigen CeSQl (CloudEvents SQL Query Language). Filter werden zu kombinierten Ausdrücken zusammengefasst – ideal für Multi-Tenant-Umgebungen.

Exact Match

Einfachster Filter. Attribut muss exakt übereinstimmen.

spec:
  filter:
    attributes:
      type: order.created
      source: shop-a

CeSQl-Ausdruck

Mächtige Filterung mit SQL-ähnlichen Operatoren (LIKE, IN, AND/OR).

spec:
  filters:
    - cesql: "type LIKE 'order%'
      AND priority = 'high'"

Kombiniert

Mehrere Dialekte kombinierbar via all: Exact-Match und Prefix gemeinsam.

spec:
  filters:
    - exact:
        team: alpha
    - prefix:
        type: order

Wo findet Filtering statt?

Consumer-Side (einziger implementierter Modus). Der Trigger liest alle Events vom Broker (Kafka Consumer Group) und filtert lokal. Events die nicht passen werden verworfen. Das bedeutet: alle Events landen im Kafka-Topic des Brokers, aber nur passende werden an den Service weitergeleitet.

Upstream Propagation (nicht implementiert): Die Idee, Trigger-Filter upstream zur Quelle zu propagieren, wurde als Design-Ziel konzipiert (KNative Issue #934), aber explizit als „wont-fix" geschlossen. Das CeSQl-Filtersystem wurde bewusst kompatibel dazu entworfen – aktuell findet Filterung jedoch ausschließlich consumer-seitig im Trigger statt.
